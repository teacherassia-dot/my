الفصل 1 (افتح/اطوِ)

الفصل 1: الرؤية الاستراتيجية والاستدامة

جوهر وحدة التبادل في الاتحادات الناشئة ليس “مشروعاً تجارياً محضاً”، بل استثماراً عاماً استراتيجياً يهدف إلى بناء سيادة إعلامية، وتخفيض كلفة الاعتماد على مصادر خارجية، وخلق قيمة مركبة للدول الأعضاء على المدى المتوسط والطويل. المنظومة التقنية هنا تُبنى لخدمة “المحتوى” و”الاستمرارية”، لا لتكديس أجهزة.

Hybrid
منهجية الجمع بين الاستقبال الفضائي + التوزيع عبر الإنترنت
SRT/RTMP
SRT للتوزيع الآمن منخفض الكمون + RTMP للتوافق
TCO / ROI
قرار تقني مبني على تكلفة الملكية والعائد التشغيلي

1.1 لماذا “وحدة التبادل” محور اقتصادي/سيادي؟

  • خفض التكاليف التشغيلية: تقليل الاعتماد على وكالات خارجية مكلفة عبر تبادل مباشر بين الأعضاء.
  • رفع الكفاءة: توحيد قنوات الاستقبال والتسجيل والفهرسة والتوزيع بدل “جزر تقنية” منفصلة.
  • تأثير مضاعف: التدريب، الإنتاج، الدعم، والفعاليات المرتبطة تُنتج قيمة اقتصادية محلية.
  • نموذج الاستدامة: منصة تبادل + خدمات تقنية + تدريب + بيع محتوى + فعاليات.

1.2 فلسفة الانتقال: من الساتلايت إلى الحلول الهجينة

الأقمار الصناعية ممتازة للاستقبال واسع النطاق، لكن التوزيع الحديث يتطلب مرونة الإنترنت: تحكم بالوصول، قياس استخدام، تسليم انتقائي للدول، وربط سريع بالمنصات. لذلك يُعتمد النموذج الهجين: Satellite Ingest → IP Processing → SRT/RTMP Distribution.

قاعدة قرار: أي خيار تقني لا يضمن “استمرارية الخدمة” و”قابلية التوسع” و”سهولة التشغيل” ضمن ميزانية محدودة هو خيار مرحلي فقط، وليس معياراً.
الفصل 2 (افتح/اطوِ)

الفصل 2: نظرة عامة على SRT/RTMP والسيناريو الهجين

2.1 لماذا SRT؟ ولماذا RTMP؟

  • SRT: مناسب لتوزيع روابط عالية الجودة للدول الأعضاء عبر الإنترنت العام، مع تشفير، تصحيح أخطاء، وكمون منخفض.
  • RTMP: مطلوب للتوافق مع بعض منصات البث/الأدوات القديمة أو تكاملات CDN تقليدية.

2.2 تعريف السيناريو الهجين “المرن”

السيناريو الهجين الموصى به في وثائق المنظومة يعتمد على: استقبال فضائي مستقر + معالجة IP مركزية + توزيع SRT/RTMP + أرشفة وفهرسة عبر MAM. جوهره تقليل عدد الأجهزة المكلفة (IRDs) قدر الإمكان مع الحفاظ على خطط بديلة (Failover).

ملاحظة تشغيلية: السيناريوهات المعتمدة على “كمبيوتر واحد” دون احتياطي تُعد مخاطرة تشغيلية؛ الحد الأدنى المقبول هو وجود جهاز احتياطي جاهز + نسخ إعدادات + آلية مراقبة واستبدال.
الفصل 3 (افتح/اطوِ)

الفصل 3: خرائط المنظومة (Maps & Flows)

3.1 المنظومة كطبقات (Layered Architecture)

  1. طبقة الاستقبال (RF/Satellite): أطباق + LNB + Multiswitch + كوابل محورية.
  2. طبقة الالتقاط/التحويل (Ingest/Transcoding): كروت TBS/أو IRD → تحويل إلى IP.
  3. طبقة الإدارة والأرشفة (MAM): فهرسة، Metadata، واجهة ويب، سياسات احتفاظ.
  4. طبقة المراقبة (Multiview/QC): متابعة القنوات وجودة الإشارة.
  5. طبقة التوزيع (SRT/RTMP): Gateways/Reflectors + تحكم وصول.
  6. طبقة الأمن والحوكمة: VLAN/QoS + جدران حماية + سجلات + صلاحيات.

3.2 مخطط كتلي مبسط

3.3 سير العمل القياسي للتبادل

  1. استقبال Feed/Channel عبر الساتلايت.
  2. تسجيل/التقاط القناة (Ingest) مع تعريف المصدر والحدث.
  3. مراقبة الجودة (Multiview + مؤشرات Signal/CPU/Bitrate).
  4. فهرسة/Metadata داخل MAM + تحديد سياسات الاحتفاظ.
  5. توزيع فوري (SRT/RTMP) للدول + روابط محكومة الصلاحيات.
  6. إخراج “حقيبة يومية/أسبوعية” + أرشفة نهائية.
الفصل 4 (افتح/اطوِ)

الفصل 4: الهندسة التأسيسية (Infrastructure)

4.1 أسس الموقع والتركيب

  • قواعد خرسانية ثابتة ومستوية، وخط رؤية خالٍ من العوائق.
  • تأريض معياري للحماية من الصواعق وربط الـ Multiswitch والراك.
  • حماية الكوابل عبر مواسير PVC ومسارات معرّفة ومرقمة.

4.2 الطاقة والتبريد

  • UPS Online Double Conversion (مثال 6kVA) لإتاحة إغلاق آمن واستمرارية قصيرة.
  • Rack 42U مع إدارة كوابل وتهوية (ومن الأفضل تبريد دقيق للسيرفرات 7/24 عند الضغط).
  • مراقبة حرارة/رطوبة + إنذار مبكر.

4.3 الشبكات (Network)

  • VLAN للبث منفصلة عن شبكة المكاتب.
  • QoS لإعطاء أولوية لتدفقات الفيديو.
  • Switch 10G (Managed) + بطاقات 10G للخوادم الحرجة.
الفصل 5 (افتح/اطوِ)

الفصل 5: المعدات والبرمجيات (Hardware & Software)

5.1 نظام إدارة المحتوى والتسجيل (MAM/Ingest)

وفق الدراسة الفنية/المالية: تم ترشيح Metus MAM + Ingest كحل اقتصادي مستقر يتوافق مع بيئة كروت TBS ويقلل الالتزامات السنوية الإلزامية مقارنة بخيارات مؤسسية أعلى كلفة.

5.2 نظام الملتي فيور (Multiviewer)

خيار برمجي مثل StreamLabs Multiscreen قادر على استقبال TS/IP مباشرة ويُعد مناسباً للميزانيات المحدودة.

5.3 التخزين (NAS/SAN)

  • NAS Enterprise 80TB+ (Synology/QNAP) حسب تقدير الدراسة.
  • سياسة احتفاظ: “خام/أولي” + “نسخة تحرير” + “أرشيف نهائي”.
  • تسمية موحدة للمحتوى + جداول Metadata.

5.4 برمجيات التشغيل المساندة

  • FFmpeg للتحويل والترميز والبث.
  • OBS/vMix عند الحاجة لإنتاج/إخراج.
  • خادم توزيع (حسب النموذج): Nginx RTMP / Nimble / MediaMTX / Live SRT Server.
الفصل 6 (افتح/اطوِ)

الفصل 6: إدارة التراخيص والبرمجيات المدفوعة (License Management)

إدارة التراخيص ليست إجراءً إدارياً ثانوياً؛ هي جزء من استمرارية الأعمال. أي ترخيص ينتهي دون متابعة قد يوقف التسجيل، أو الأرشفة، أو البث، أو الدعم الفني في وقت حساس. لذلك نعتمد سجل أصول برمجية (Software Asset Register) مع مسؤوليات واضحة وتذكير مبكر.

6.1 استراتيجية التراخيص: Perpetual vs Subscription

  • Perpetual (شراء لمرة واحدة): يقلل OPEX الإلزامي ويُناسب الميزانيات المحدودة، مع تحديثات اختيارية.
  • Subscription/SaaS: يقلل عبء البنية التحتية المحلية لكنه يفرض OPEX ثابتاً ويعتمد على الدفع المستمر.

6.2 الميزانية التشغيلية (Software OPEX)

تُحتسب سنوياً: دعم/تحديثات (SLA) + تراخيص VPS/Streamer + أي Plugins مدفوعة + شهادات/مفاتيح تشفير عند اللزوم. الهدف: منع المفاجآت عبر وضع بند ثابت في الميزانية التشغيلية.

6.3 نموذج سجل التراخيص (جدول متابعة)

البرنامج/الخدمة نوع الترخيص الاستخدام/الدور تاريخ البدء تاريخ الانتهاء/التجديد الكلفة السنوية (OPEX) المسؤول مستوى الأهمية ملاحظات امتثال/بدائل
Metus MAM + Ingest Perpetual + SLA اختياري تسجيل/فهرسة/أرشفة + Web ____ تحديثات: سنوي (اختياري) ~4,000$ (اختياري) مدير الأنظمة / مسؤول البرمجيات حرج بديل أعلى كلفة: Cinegy (SLA إلزامي)
StreamLabs Multiscreen Perpetual / حسب العرض Multiview/QC ____ ____ ____ مهندس غرفة التحكم عالي بديل مكلف: TAG VS
VPS/Streaming License (مثال: MENOS / Nimble Panel) Subscription Gateway/Reflector للبث ____ شهري/سنوي حسب الخطة مسؤول الشبكات حرج بديل مفتوح المصدر: MediaMTX/Live SRT (يتطلب خبرة)
vMix / Plugins (عند الحاجة) Perpetual / Subscription إنتاج/إخراج ____ ____ ____ مسؤول الإنتاج التقني متوسط بديل مجاني: OBS (مع قيود)

6.4 ضوابط الامتثال

  • يُمنع تشغيل أي برنامج “غير أصلي/غير مرخص” على خوادم البث والأرشفة.
  • حفظ مفاتيح التراخيص في مخزن آمن (Password Manager/Sealed Envelope) وتحديد صلاحيات الوصول.
  • مراجعة ربع سنوية لسجل التراخيص + تنبيه قبل 60/30/7 يوم من الانتهاء.
قاعدة تشغيل: الترخيص الذي يوقف خدمة حرجة يجب أن يكون له (بديل جاهز) أو (خطة تمديد مؤكدة) قبل 30 يوماً على الأقل.
الفصل 7 (افتح/اطوِ)

الفصل 7: ميثاق العمل الهندسي وتطوير الكادر البشري (Engineering Charter)

طبيعة الميثاق: هذا الفصل مكتوب بصياغة إدارية ملزمة، ويُعد مرجعاً للتكليف والمساءلة والتحسين المستمر.

7.1 الغرض والنطاق

يحدد هذا الميثاق مسؤوليات الإدارة الهندسية في وحدة التبادل، وحدود التداخل مع إدارة الأخبار/المحتوى، وآليات التشغيل، والتصعيد، وقياس الأداء. الهدف هو ضمان أن تعمل المنظومة 7/24 بأعلى استقرار ممكن ضمن الموارد المتاحة.

7.2 مبادئ حاكمة (Principles)

  1. التقنية لخدمة الإبداع: دور الهندسة هو تبسيط وأتمتة الأنظمة لتفريغ المنسقين والمحررين لإنتاج قيمة محتوى.
  2. الاستمرارية قبل الرفاهية: أي إضافة تقنية لا تُحسن الاستقرار أو قابلية التشغيل تُؤجل.
  3. التدرج المعياري: البناء Modular يسمح بالنمو دون إعادة تأسيس.
  4. الشفافية التشغيلية: كل عطل يُوثق، وكل تغيير يُسجل، وكل قرار يُراجع.

7.3 المسؤوليات الأساسية للإدارة الهندسية

  • تشغيل المنظومة: استقبال، تسجيل، مراقبة، توزيع، وأرشفة وفق إجراءات تشغيل معيارية (SOP).
  • الصيانة الوقائية: تبريد، UPS، كوابل، تحديثات نظام، فحوصات دورية للإشارة.
  • إدارة التغيير (Change Management): لا تغيير على الخوادم أو الشبكة دون تذكرة تغيير مع اختبار رجوع (Rollback).
  • إدارة الأصول البرمجية: سجل تراخيص، نسخ إعدادات، مستودع أكواد/سكربتات، وتحديثات مُدارَة.
  • الدعم والتدريب: تدريب المنسقين والصحفيين على الاستخدام الصحيح للمنصات لتقليل الأخطاء وتسرّع الاستجابة.

7.4 العلاقة مع إدارة الأخبار/المحتوى (RACI مبسط)

المهمة مسؤول (R) معتمد (A) مستشار (C) مُبلّغ (I)
تشغيل الاستقبال والتسجيل (Ingest) الهندسة مدير الهندسة المحتوى الإدارة
تحديد أولويات التبادل وجدولة المحتوى المحتوى مدير وحدة التبادل الهندسة الدول الأعضاء
إتاحة روابط SRT/RTMP وإدارة الوصول الهندسة مدير الهندسة الأمن/الإدارة المحتوى/الأعضاء
نشر سياسة الأرشفة والتسمية الهندسة + الأرشفة الإدارة المحتوى الجميع

7.5 واجبات الأفراد والانضباط التشغيلي

  • أي خلل يُبلّغ عبر قناة معتمدة (تذكرة/نموذج) مع وقت ومصدر وخطوات إعادة الإنتاج.
  • يُمنع تغيير إعدادات السيرفر/الشبكة دون إذن وتوثيق.
  • يلتزم الجميع بسياسة كلمات المرور، وعدم مشاركة الحسابات.
  • الالتزام بحدود الصلاحيات (Least Privilege) وتجنب “الحلول السريعة غير الموثقة”.

7.6 تطوير الكادر: خارطة تدريب مختصرة مرتبطة بالمنظومة

(يُستكمل بالتفصيل وفق خطة تطوير الكوادر 2025–2027: شبكات، أمن، أرشفة، SMPTE 2110، تشغيل غرف التحكم، DRills الطوارئ…)

  • شهر 1–2: أساسيات غرفة التحكم + Troubleshooting Under Pressure.
  • شهر 3–6: شبكات (VLAN/QoS/Multicast) + أمن أساسي.
  • شهر 6–12: MAM/أرشفة + أتمتة Python + مراقبة SNMP.
خلاصة الميثاق: المنظومة لا تُدار بالأجهزة وحدها؛ تُدار بعقلية تشغيل منضبطة، وتوثيق، وتدريب، ومسؤوليات واضحة.
الفصل 8 (افتح/اطوِ)

الفصل 8: التشغيل والصيانة (Operations)

8.1 إجراءات التشغيل القياسية (SOPs) — الحد الأدنى

  1. فحص جودة الإشارة (Signal Quality ≥ 70%) قبل أي حدث.
  2. تأكيد جاهزية التخزين ومساحة NAS.
  3. فحص البث (SRT/RTMP) عبر نقطة اختبار داخلية وخارجية.
  4. تفعيل المراقبة (SNMP/Logs) قبل التشغيل.
  5. توثيق كل بث: المصدر، الوقت، الدولة المستفيدة، نوع الحقوق.

8.2 صيانة وقائية (نموذج مختصر)

النشاط التكرار الوقت المسؤول
فحص جودة الإشارة آلياً يومي مستمر نظام مراقبة
تنظيف فلاتر التبريد أسبوعي 15 دقيقة فني صيانة
نسخ احتياطي كامل (Image) أسبوعي 2 ساعة نظام آلي
اختبار Failover شهري 30 دقيقة مهندس النظام
الفصل 9 (افتح/اطوِ)

الفصل 9: الأمن السيبراني (Cyber Security)

9.1 مبدأ الدفاع متعدد الطبقات (Defense in Depth)

  • فصل الشبكات (VLAN/Segmentation) + جدار حماية.
  • تقييد المنافذ والبروتوكولات (SRT/UDP ports) حسب الحاجة فقط.
  • حسابات منفصلة، MFA حيث أمكن، ومبدأ أقل صلاحية.
  • سجلات مركزية (Syslog) + مراقبة SNMP/Alerts.

9.2 سياسات إلزامية مختصرة

  • لا إدارة عن بعد بدون VPN/قناة مؤمنة.
  • تحديثات أمنية مجدولة واختبارية قبل الإنتاج.
  • خطة استجابة للحوادث: عزل → تحويل للاحتياطي → تحليل → استعادة.
الفصل 10 (افتح/اطوِ)

الفصل 10: درء الكوارث واستمرارية الأعمال (DR/BCP)

10.1 أهداف الاستعادة (RTO/RPO)

  • RTO: زمن العودة للخدمة (هدف واقعي: دقائق للبث، ساعات للأرشفة حسب السيناريو).
  • RPO: مقدار البيانات المسموح فقدها (هدف: إعدادات يومية + Image أسبوعي).

10.2 مكوّنات DR الأساسية

  • جهاز احتياطي Standby بمواصفات مماثلة.
  • كرت TBS احتياطي/بديل أو عدد محدود من IRDs للقنوات الحرجة.
  • نسخ إعدادات تلقائي (rsync) + Repository Git للسكريبتات.
  • UPS مزدوج (أو 2N تدريجياً) + مزودا إنترنت عند الإمكان.

11.2 OPEX سنوي (نموذج تقديري)

  • SLA/Updates (اختياري حسب السياسة) + استضافة VPS إن وجدت + صيانة/قطع غيار + إنترنت.
  • مبدأ الترشيد: اختيار تراخيص Perpetual حيث يكون الأثر التشغيلي عالياً وتكاليف الاشتراك مرهقة.
ملاحظة مالية: القرار الصحيح هو الذي يخفض TCO ويمنع “الرسوم الإلزامية” ما أمكن، مع الحفاظ على دعم تقني حدّه الأدنى للمكونات الحرجة.
الفصل 12 (افتح/اطوِ)

الفصل 12: الخاتمة والتوصيات

التأسيس الصحيح لوحدة التبادل يبدأ بتحديد “النموذج التشغيلي” قبل شراء الأجهزة: من هو المسؤول؟ ما هي إجراءات التشغيل؟ كيف نقيس الأداء؟ وكيف نضمن الاستمرارية؟ النموذج الهجين (Satellite + IP + SRT/RTMP) هو الأكثر واقعية للاتحادات الناشئة لأنه يوازن بين الاستقرار والميزانية والمرونة.

توصيات تنفيذية مختصرة

  • البدء بتأسيس البنية (Rack/UPS/Network/VLAN/QoS) قبل توسعة القنوات.
  • اعتماد MAM اقتصادي مستقر (Metus) لتقليل OPEX الإلزامي.
  • تنفيذ DR من اليوم الأول: جهاز احتياطي + نسخ إعدادات + اختبار شهري.
  • استثمار التدريب المستمر: الشبكات، الأمن، الأرشفة، الأتمتة.
  • تطبيق ميثاق العمل الهندسي كوثيقة تشغيل ومساءلة وتحسين.
ملحق A (افتح/اطوِ)

ملحق A: قاموس مصطلحات مختصر

  • Ingest: إدخال/التقاط الإشارة للتسجيل والمعالجة.
  • MAM: إدارة الأصول الإعلامية (Metadata, Search, Archive).
  • Multiview: عرض متعدد لمراقبة عدة قنوات في وقت واحد.
  • SRT: بروتوكول نقل آمن وموثوق منخفض الكمون.
  • RTMP: بروتوكول بث قديم نسبياً للتوافق مع منصات عديدة.
  • DR/BCP: درء الكوارث/استمرارية الأعمال.
  • CAPEX/OPEX: نفقات رأسمالية/تشغيلية.
